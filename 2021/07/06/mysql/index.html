MySQL的事务

什么是事务？

简介: 事务是作为单个逻辑工作单元执行的一系列操作。一个逻辑工作单元必须有四个属性，称为原子性、一致性、隔离性和持久性 (ACID) 属性，只有这样才能成为一个事务。事务一般都是与数据库打交道的操作

事务特性ACID

  • Atomicity（原子性）：一个事务必须被视为一个不可分割的最小工作单元，整个事务中的所有操作要么全部提交成功，要么全部失败回滚，对于一个事务来说，不可能只执行其中的一部分操作。
  • Consistency（一致性）：数据库总是从一个一致性状态转换到另一个一致状态。下面的银行列子会说到。
  • Isolation（隔离性）：通常来说，一个事务所做的修改在最终提交以前，对其他事务是不可见的。注意这里的“通常来说”，后面的事务隔离级级别会说到。
  • Durability（持久性）：一旦事务提交，则其所做的修改就会永久保存到数据库中。此时即使系统崩溃，修改的数据也不会丢失。（持久性的安全性与刷新日志级别也存在一定关系，不同的级别对应不同的数据安全级别。）

为了能更好的理解事务，下面以银行转账为例: 

START TRANSACTION;
SELECT balance FROM checking WHERE customer_id = 10233276;
UPDATE checking SET balance = balance - 200.00 WHERE customer_id = 10233276;
UPDATE savings SET balance = balance + 200.00 WHERE customer_id = 10233276;
COMMIT;
  • 原子性：要么完全提交（10233276的checking余额减少200，savings 的余额增加200），要么完全回滚（两个表的余额都不发生变化）
  • 一致性：这个例子的一致性体现在 200元不会因为数据库系统运行到第3行之后，第4行之前时崩溃而不翼而飞，因为事务还没有提交。
  • 隔离性：允许在一个事务中的操作语句会与其他事务的语句隔离开，比如事务A运行到第3行之后，第4行之前，此时事务B去查询checking余额时，它仍然能够看到在事务A中被减去的200元（账户钱不变），因为事务A和B是彼此隔离的。在事务A提交之前，事务B观察不到数据的改变。
  • 持久性：当整个流程执行完毕以后我们的数据会永久存储起来，不会在下次访问的时候不存在。

上面提到了事务的隔离性那么在Mysql 中事务的隔离总共有4个级别：

  • 读未提交（Read Uncommitted）
  • 读已提交（read committed）
  • 可重复读 (Repeatable Read)
  • 串行化 （serializable）

事务的隔离级别是从上往下依次增加的。性能恰恰相反。采用哪种隔离界要根据具体的系统需求来权衡决定，其中[可重复读]是MySQL默认的隔离级别。

为什么需要隔离级别呢？

事务的隔离级别说白了就是为了解决，脏读、可重复读、幻读的。

    脏读：
        当多个事务同时在一个进程中执行的时候，如果当前事务读到了其他事务未提交的数据，那么就会产生脏读的情况，未提交意味着可能这个数据需要回滚，也就是可能最终这个数据不会存到数据库中，也就是不存在的数据。读到了这样的数据，那就是脏数据。
    可重复读：
        可重复读指的是在一个事务内，最开读到的数据和事务结束前的任意时刻读到的是同一批数据。
    不可重复读： 
        对比可重复读，不可重复读的概念就是在一个事务中开始于结束读到的数据可能不是同一批数据，比方说其他事务提交了数据。
    幻读：    
        幻读一般指，当一个事务插入了一条数据，这个时候另外一个事务select 这个数据的是是查询不到的，但是当另外一个事务插入同一个数据的时候，会提交数据不可重复插入。这就是幻读。

mysql 解决幻读的方式[MVCC]

什么是MVCC呢？
就是多版本并发控制。MVCC 是一种并发控制的方法，一般在数据库管理系统中，实现对数据库的并发访问。在我看来，他无非就是乐观锁的一种实现方式。
MVCC的实现是通过保存数据在某一个时间点快照来实现的。也就是说不管实现时间多长，每个事物看到的数据都是一致的。

MVCC 是如何工作的

InnoDB的MVCC是通过在每行记录后面保存两个隐藏的列来实现。这两个列一个保存了行的创建时间，一个保存行的过期时间（删除时间）。当然存储的并不是真实的时间而是系统版本号（system version number）。每开始一个新的事务，系统版本号都会自动新增。事务开始时刻的系统版本号会作为事务的版本号，用来查询到每行记录的版本号进行比较。

一般在并发中我们大多是通过互斥锁 for update 来锁住记录来处理请求的。但是MVCC 是通过不用加锁来做到的。
其实中mysql 中读这个操作分为俩个，一个是 [快照读] ， 一个是当前读。

  • 当前读： 它读到的数据都是当前最新的版本数据，会对当前读取到的数据进行加锁，防止其他事务修改数据，这其实是一种悲观锁的操作.
    • select lock in share mode (共享锁)
    • select for update (排它锁)
  • 快照读： 快照读是基于多版本并发空值的实现，也就是所谓的MVCC。既然是多版本，也就是说快照读到的数据不一定是最新的数据，有可能是历史数据。

数据库并发的场景

读-读 ：不存在任何问题，也不需要并发控制
读-写 ：有线程安全问题，可能会造成事务隔离性问题，可能会产生，脏读，幻读，不可重复读的问题。
写-写 ：有线程安全问题，可能更新的数据丢失。

MVCC 解决并发的哪些问题？

mvcc 可以解决 脏读、幻读、不可重复读等事务问题，但是这些都是基于 读-写的时候才可以。对于，写-写的问题，mvcc 是解决不了的。

所以需要组合的方式来解决：

  • MVCC + 悲观锁

Mvcc 的实现原理

它的实现原理其实主要就是通过 版本链，undo日志，read view 来实现的。

  • 版本链：
    数据库中的每行数据，除了我们肉眼能看见的数据以外，还有几个隐藏的列，分别是 [db_trx_id],[db_roll_pointer],[db_row_id]
    • db_trx_id ： 这个字段总共有6 byte ,它是最近修改、插入的事务ID.也就是说记录了这条记录，最后一次修改该记录的事务ID。
    • db_roll_pointer 这个字段总共有7 btye ,指向的是是上一个版本的地址，也就是指向undolog 里边的一条记录。
    • **db_row_id :**这个字段 6 byte 如果没有主键 innodb 引擎会自动以它来产生一个聚簇索引。

undo 日志

  • 保证事务 rollback 时的原子和一致性，当事务进行回滚的时候可以用undo log 的数据进行恢复。
  • 用于MVCC 快照读的数据，在MVCC 多版本控制中，通过读取undo log 的历史记录数据可以实现不同的事务版本号，都会拥有自己独一无二的数据版本。

undolog 分为俩种

  • 一种是插入 undo log : 即事务插入一条新记录的时候就会产生一条undo Log 日志，只在事务回滚的时候需要，但是当事务提交以后这条日志将被清除。

  • 更新 undo log : 事务在进行Update 或者是del 的时候产生的日志；这条日志不仅在回滚时候需要，在快照读的时候也需要。所以不会被删除。
